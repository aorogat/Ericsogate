Constructing Knowledge Graphs from Heterogeneous Data Sources

Use Case in Ericsson

Ericsogate is a modular, six-layer framework that transforms isolated data from relational databases, JSON files, and text logs into a unified Knowledge Graph for Cloud Radio Access Networks (CloudRAN).

By Abdelghny Orogat, Sri Lakshmi Vadlamani, Dimple Thomas, and Ahmed El-Roby

Understanding Ericsogate

A comprehensive guide to how Ericsogate transforms isolated data sources into an intelligent, interconnected Knowledge Graph

What Problem Does Ericsogate Solve?

In Cloud Radio Access Networks (CloudRAN), data comes from everywhere—relational databases, JSON files, text logs, system metrics. Traditionally, this data exists in silos. A test case might be stored in one database, its execution results in another, and the software version details in yet another system. When engineers need to understand why a test failed, they must manually piece together information from multiple sources—a time-consuming and error-prone process.

Ericsogate solves this by creating a Knowledge Graph—a web of interconnected data where relationships are explicit. Click on a failed test, and instantly see the linked software version, the engineer who ran it, and the specific hardware configuration used. What took hours now takes seconds.

How Knowledge Graphs Work

From Isolated Data to Connected Intelligence

Imagine three separate data sources:

  • Relational Database: Contains cell tower data (cell_id: 123, bandwidth: 20MHz, location: Ottawa)
  • JSON File: Has technical specifications (cell_id: 123, technology: 5G)
  • Text Document: Contains location information ("Ottawa has a population of 1 million...")

In traditional systems, these remain disconnected. But Ericsogate recognizes that cell_id 123 and Ottawa appear in multiple sources. It creates a unified graph where:

  • Cell 123 → uses → 5G technology
  • Cell 123 → has bandwidth → 20MHz
  • Cell 123 → located in → Ottawa
  • Ottawa → has population → 1M

Now you can traverse relationships: starting from a cell tower, follow connections to find its location, technology, and even demographic data about that location—all from a single query.

RDF Data Model and Ontology

The Building Blocks: Instance Data and Ontology

Knowledge Graphs consist of two essential components:

Instance Data

The detailed, factual information about specific entities. For example, "TestCase_42 ran on Server_A and failed with error code 500." This represents individual facts from your data sources—test results, cell tower configurations, software versions.

Ontology

The schema that defines classes and their relationships. It starts as a shared structure for data integration but evolves as developers enrich it. For example, the ontology defines that "Functional Test" and "Performance Test" are both subclasses of "Test Case," allowing you to query all tests regardless of their specific type.

How they connect: Instance data links to the ontology through type predicates. Without the ontology, test cases (TC1, TC2) exist as isolated facts. With the ontology, they connect through their classes, enabling you to find all test runs across different data sources by following the class hierarchy.

RDF Triples: Data is represented as <subject, predicate, object> triples. For example: <TestCase_42, hasResult, "Failed"> or <Cell_123, locatedIn, Ottawa>. These triples form an interconnected network where subjects and objects are nodes, and predicates are edges defining relationships.

The Six-Layer Architecture

Each layer has a specific responsibility, working together to transform raw data into actionable intelligence

Figure 1: Ericsogate's comprehensive modular architecture for CloudRAN data management.
Layer 1

Ingestion Layer: Gathering Data from Everywhere

What it does: This layer connects to all your data sources—databases, APIs, log files, JSON documents—and pulls in raw data.

How it works:

  • Raw Data Readers fetch data from each source using APIs or direct connections
  • Parsers interpret the data structure—whether it's a database table, JSON object, or unstructured text
  • Data Cleaning fixes inconsistencies (removing duplicates, correcting misspellings)
  • Data Validation ensures data meets quality standards before moving forward
  • Origin Trace Readers record metadata: where did this data come from? when? who created it?

Key benefit: Ensures data provenance and quality. You always know where your data came from and can trust its accuracy.

Layer 2

Transformation Layer: Standardizing the Chaos

What it does: Converts all different data formats into a single, standardized JSON format that represents graph relationships.

How it works:

  • Each data source has a JSON Transformer that converts its specific format
  • Data becomes triples: Subject-Predicate-Object (e.g., "Cell_123" "hasLocation" "Ottawa")
  • Local Ontologies define the class hierarchy for each source (e.g., "Functional Test" is a subclass of "Test Case")
  • Developers work with familiar JSON—no need to learn RDF or complex graph formats

Example: A database row becomes: {"subject": "TestCase_42", "predicate": "hasResult", "object": "Failed"}

Key benefit: Developers use standard JSON tools and libraries—dramatically reducing the learning curve.

Layer 3

Control Layer: The Orchestra Conductor

What it does: Orchestrates the entire data pipeline—scheduling updates, aligning entities, and ensuring everything runs smoothly.

How it works:

  • Ingestion Registry keeps track of all data sources and their refresh rates
  • Scheduler automatically requests fresh data at configured intervals (from milliseconds to days)
  • Entity Alignment identifies when different sources refer to the same thing:
    • Local Matching: "cell_123" and "Cell 123" are the same entity
    • Global Matching: "Ottawa" in the database matches "Ottawa" in the text document
  • JSON-RDF Transformer converts the standardized JSON into RDF triples for storage
  • Global Ontology links class hierarchies across data sources, creating connections

Key benefit: Automatic data updates and intelligent entity matching—no manual data integration needed.

Layer 4

Data Management Layer: Scalable Storage Engine

What it does: Stores the Knowledge Graph and handles queries, scaling horizontally across multiple servers as data grows.

How it works:

  • Uses Apache Jena (open-source triple store) to store RDF data
  • DM Controller distributes data across multiple graph engines when a single server isn't enough
  • Executes Federated SPARQL Queries across distributed engines
  • Presents a single-graph facade—applications see one unified graph, not multiple databases

Scalability Example: When your graph reaches 100 million triples, the controller automatically distributes it across 5 servers. Queries run in parallel, and results merge seamlessly.

Key benefit: Unlimited storage capacity through horizontal scaling—add more machines as data grows.

Layer 5

API Layer: The Security Gateway

What it does: Provides secure, easy-to-use REST APIs that return JSON data—hiding all graph database complexity.

How it works:

  • Endpoint Manager dynamically creates API endpoints for different applications
  • Request Handler translates simple API requests into complex SPARQL queries
  • Returns results in standard JSON format
  • Access Control: Each endpoint has unique credentials—users only access authorized data
  • Applications can combine data from multiple endpoints for comprehensive queries

Developer Experience: Instead of writing SPARQL, developers call: GET /api/test-cases?status=failed

Key benefit: Application developers need zero knowledge of SPARQL or RDF—they work with familiar REST APIs.

Layer 6

Application Layer: Where Intelligence Becomes Action

What it does: Houses user-facing applications that leverage the Knowledge Graph for decision-making.

Applications include:

  • Semantic Search Dashboards for exploring interconnected data
  • KG Summarization Tools that condense 100,000 nodes into 100 meaningful insights
  • Recommendation Systems that suggest related tests or configurations
  • Error Detection Systems that identify patterns across failures
  • Visualization Tools for exploring the knowledge graph interactively

Key benefit: Rapid application development—focus on business logic, not data integration.

The Complete Data Journey

Here's how data flows through all six layers:

  1. Collection: Layer 1 pulls test results from a database every hour
  2. Standardization: Layer 2 converts them to JSON triples with local ontology
  3. Orchestration: Layer 3 aligns "Test_42" with "TestCase_42" from another source, merges with global ontology
  4. Storage: Layer 4 stores the unified graph, distributing across servers as needed
  5. Access: Layer 5 exposes endpoint: /api/tests/42 with proper authentication
  6. Insight: Layer 6 dashboard shows Test 42's failure, linked to software version 3.2.1, run by Engineer Jane on Server A

Result: What used to require querying 5 different systems and manually correlating data now happens instantly with a single click.

Industrial Applications

Decision Support

KG Summarization

Instead of overwhelming stakeholders with 100,000+ nodes, our Pattern-Statistics-based Method provides a concise view (approx. 100 nodes) while maintaining 94% representativeness. This allows managers to see test case trends and outcomes at a glance.

How it works: The system abstracts individual instances by their class types. For example, instead of showing TestCase_1, TestCase_2, TestCase_3 individually, it shows "TestCase" with statistics: "3 test cases, executed 4 times, 2 functional, 1 performance."

Interlinked Querying

Semantic Search

A dashboard that allows users to interrogate data based on meaning rather than keywords. Users can click on a failed test run and instantly see the specific software version and developer logs linked in the KG.

Real-world impact: What previously required manually querying multiple systems and correlating results now happens with a single click, reducing investigation time from hours to seconds.

Summarization in Action

The Challenge: A knowledge graph with 100,000+ nodes representing test cases, test runs, software versions, and configurations is too complex for stakeholders to understand.

Traditional Approaches Fall Short:

  • Grouping-based methods merge nodes into supernodes, losing critical distinctions
  • Bit compression reduces storage but doesn't facilitate analysis
  • Simplification techniques remove nodes/edges but may miss important patterns

Ericsogate's Solution: The Pattern-Statistics-based Method leverages the KG's ontology structure. Instead of showing individual test runs, it displays:

  • TestCase class (3 instances total)
  • → Functional Test subclass (2 instances, ran 2 times)
  • → Performance Test subclass (1 instance, ran 1 time)
  • Total test runs: 4

Result: Stakeholders see meaningful patterns at a glance—which test types are most common, how often they run, and where failures concentrate—without drowning in raw data.

Detailed Performance Analysis

Summarization Performance

Compression Rate: Ericsogate achieves 99.90% compression, reducing 100,000 nodes to approximately 100 summary nodes while maintaining 94% representativeness.

Scalability: The system scales horizontally—adding more classes increases summary size linearly—but remains unaffected by vertical scaling (more entities per class).

Semantic Search Performance

Update Speed: Modifying search features that previously took 20H hours now takes H hours—a 20x improvement.

Feature Development: The team can now add 10 times more features per month, accelerating innovation and user value delivery.

Lessons Learned: KG Mastery is Optional

Our framework abstracts SPARQL and RDF complexities. Ingestion developers work with standard JSON, while application developers use clean API endpoints, reducing the learning curve and accelerating team integration.

Key Insight: Only developers working on the Control and API layers need SPARQL knowledge. All other team members—data ingestion developers, application developers—work with familiar technologies (JSON, REST APIs), dramatically reducing training time and enabling faster onboarding.

Deployment Architecture

Cloud-Native Infrastructure: Ericsogate leverages Kubernetes and Helm for containerized deployment, ensuring:

  • High Availability: Pods can be replicated across multiple servers
  • Fault Tolerance: Automatic recovery from failures
  • Elastic Scaling: Resources scale automatically based on demand
  • Independent Deployment: Each layer can be updated without affecting others

Pod Configuration:

  • Application Pods: House API endpoints and data readers
  • Control Layer Pod: Manages scheduling and data ingestion
  • Data Management Pods: Distributed triple stores for horizontal scaling
